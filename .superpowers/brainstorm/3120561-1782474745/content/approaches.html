实现方案对比

三种架构分解策略 — 点击选择你倾向的方案

A

水平切分（按功能）

每个子 spec 是一个完整的垂直功能（会话列表、Agent输出、文件Tab等），完成即可独立使用。

顺序： 基础设计系统 → 布局壳 → [会话管理 | Agent工作区 | 文件Tab | 右侧边栏 | 设置面板] 并行
优点： 每个交付物是可用功能
缺点： 需先完成共享布局/主题基础
B

垂直切分（按层）

先做所有 UI 外壳，再接入数据层，最后加交互逻辑。

顺序： 所有UI壳 → 所有数据集成 → 所有交互逻辑
优点： 关注点分离清晰
缺点： 全部层完成后才有可用功能，反馈周期长
C

混合切分（基础 + 功能切片）⭐ 推荐

先构建基础（设计系统 + 布局壳），再以功能切片并行开发。

顺序： 基础+布局 → [会话管理 ∥ Agent工作区 ∥ 文件Tab ∥ 右侧边栏 ∥ 设置面板] 并行
优点： 坚实基础使后续并行开发更一致；每个切片可独立验证
缺点： 基础阶段是阻塞性的，但耗时短
推荐方案 C 的理由：
1. 基础设计系统（shadcn + 主题 + i18n）是所有组件的依赖，必须先完成
2. 布局壳（dockview 集成 + 三栏布局）是所有面板的容器，也需先完成
3. 这两个基础 spec 完成后，5 个功能切片可以并行开发，大幅缩短总工期
4. 每个切片有清晰的验收标准，适合用 *-dev skill 独立执行
